# 判据：用户要求 vs 运行记录

逐段判断 `memory.md` / `*REQUIREMENTS*.md` 里每个单元属于哪一类。判据是**通用**的，不依赖任何具体项目的词汇。

## 三个标签

- `requirement` —— **持久的用户意图**：描述「以后每次都该怎么做」的偏好、规则、约束、输出标准、收尾步骤。它与某一次具体运行无关，换到下一天/下一题依然成立。
- `record` —— **某一次运行的过程或结果**：发生在特定时间/特定输入上的状态、产出、回执。只用于事后追溯，不该作为下一轮 Agent 的指令。
- `drop` —— 噪声、与任何东西都无关的残留、或与已有要求**完全重复**的内容。

## requirement 的信号

- 祈使/规范语气：必须 / 务必 / 禁止 / 不要…… / 应该 / 优先 / 永久偏好 / 以后每次；always / never / must / should / prefer。
- 描述**期望行为或标准**，而非某次的发生：输出格式/长度/分段、命名规范、禁止用某工具、固定收尾步骤等。
- 不含某一次运行专有的日期、记录 ID、消息回执 ID、产出文件名。

## record 的信号

- 出现具体日期/时间戳，或某次运行专有的不透明 ID（记录 ID、消息回执 ID 等）。
- 陈述**已发生**的事：本次处理的是 X、产物保存为 Y、通知发送成功 Z、运行起止时间、本轮跳过/处理了哪些。
- 即使含「成功/失败/状态」字样，只要绑定到某一次运行，就是 record。

## 边界情形：一段里既有要求又有记录

最常见也最关键。抽象形态（用占位符表示，真实内容随项目而异）：

> 「用户要求〈某产物〉必须满足〈某标准〉。本次已据此修正〈某次产物〉，旧版备份在〈路径〉，通知回执〈ID〉。」

处理方式：**拆开**。
- 把其中「〈某产物〉必须满足〈某标准〉」这条**持久规则**作为一个 `requirement`（在 classification.json 里用 `text` 字段给出抽取后的纯规则文本）。
- 其余「修正了什么、备份在哪、回执 ID」作为 `record`。

在 classification.json 里可以让同一个 `id` 出现两次：一条 `requirement`（带 `text` 覆盖为提炼后的规则），一条 `record`。

## 去重（重要）

把某条 `requirement` 加进 automation.toml 之前，**先对照现有 prompt**：如果同义内容已经在 prompt 里，标记为 `drop`，不要重复加入。吐槽文里「同样的要求提了两次」正是因为没做这一步。

## classification.json 的格式

两种皆可：

```json
{ "u0007": "requirement", "u0008": "record", "u0009": "drop" }
```

或（支持拆分与文本覆盖）：

```json
[
  { "id": "u0012", "label": "requirement", "text": "〈从该单元提炼出的纯规则文本，去掉本次运行的具体细节〉" },
  { "id": "u0012", "label": "record" },
  { "id": "u0013", "label": "record" }
]
```
