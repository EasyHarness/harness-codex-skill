# 结构化 prompt 骨架

`automation.toml` 的 `prompt` 往往是一坨没有结构的长字符串，要求、流程、输出标准、收尾步骤混在一起，新要求只能往后贴。重建时把它重写成清晰分节、去重的结构。这是**通用骨架**，按实际内容取舍，不要硬塞空节。

## 推荐分节

```
# 任务目标
一句话说明这个 automation 要干什么、产出什么。

# 触发与范围
何时运行、处理哪些输入、时区/日期口径、什么情况下应停止并报告阻塞。

# 硬性规则
逐条、可执行、去重。把散落在 memory/REQUIREMENTS 里的持久要求收敛到这里。
1. ……
2. ……

# 数据与工具
允许/禁止使用的工具、数据源、凭据约束、固定的 CLI/接口与已验证的常量
（如目标 chat_id 等运行所需的固定事实，应保留）。

# 输出要求
格式、长度、分段、命名、保存位置等成品标准。

# 收尾与通知
每次运行结束固定要做的事（报告字段、通知渠道、附件等）。
```

## 重写原则

- **去重**：合并同义条目；同一约束只出现一次。删掉自相矛盾的旧便利贴，保留最新口径。
- **一条一句**：硬性规则用编号列表，每条独立、祈使、可验证。
- **保留运行所需的固定事实**：已验证的 ID、入口链接、固定命令属于「数据与工具」，不是运行记录，要留。
- **剔除运行记录**：任何「某次产物/某天输入/某条回执」都不进 prompt，应已被 route_md 归档到 run-history.md。
- **不引入新规则**：只重组与合并已有要求 + 本次审核通过的新增要求；不要自己发明约束。
- **保持语言**：与原 prompt 一致（中文项目就用中文）。

## 落盘

把重写好的完整 prompt 写到一个纯文本文件（如 `new_prompt.txt`），交给：

```
restructure_toml.py --toml <automation.toml> --prompt-file new_prompt.txt   # 先 dry-run 看 diff
restructure_toml.py --toml <automation.toml> --prompt-file new_prompt.txt --apply
```

脚本默认用 TOML 多行字符串写回（`"""…"""`），让 toml 文件本身可读；会自动备份并做写回后解析校验。
